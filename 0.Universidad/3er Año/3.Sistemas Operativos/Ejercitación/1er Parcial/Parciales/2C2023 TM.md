# Teoría
1. 
   Para poder procesar las interrupciones, es necesario relaizar un cambio de contexto, pero puede darse el caso que no sea necesario hacer un cambio de modo. Un ejemplo puede ser cuando el planificador por biblioteca de un KLT implemente un algoritmo de RR y que este posea varios hilos ULT, cuando ocurra un corte en el flujo de cpu en uno de los ULT (por la interrupción por fin del quantum), no será necesario realizar un cambio de modo, solo de contexto entre ese hilo y el que le siga. 
2. 
   Se podría crear un proceso por cada petición recibida, dando así libertad para poder eliminarla y que, en caso de que ocurra un fallo con la misma, no afecte a las demás peticiones entrantes. El problema con esto sería que es mucho más costos crear nuevos procesos que nuevos hilos, por lo que habría más tiempo de overhead para la creación de las estructuras de estos. 
3. 
   FIFO: Mínimo, NO, SI
   Prioridades(con desalojo): Médio(depende de la disperción con la que llegan los procesos con diferentes prioridades), SI, SI
   RR: Médio(depende del Q), NO, NO

   b. Falso. Tambien es necesario que existan varias CPUs para que se pueda apreciar la diferencia de performance en dicha situación (no basta con que la región crítica sea corta)
